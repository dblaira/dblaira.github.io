Reverse-spec extract (derived from compositions above)
Implied components
→ CapacityInput — 3-dim self-rating
→ MoveCard — action + category + chain
→ MoveQueue — ranked list with filtering
→ CausalChain — ontology path rendered as text
→ Move type in types.ts
→ rankMoves() scoring function
Key design decisions
1. How many capacity dims? (1 vs 3)
2. Score visible or hidden?
3. Show filtered moves or hide them?
4. Focus-one vs ranked-list layout?
5. Where do moves come from? (hardcoded seed vs user-defined)
Center of gravity
The causal chain is the product. The move list is the delivery mechanism. The capacity input is the filter.

"sleep → exercise → affect +2d" is worth more than any design chrome. It makes the ontology tangible.
